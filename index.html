Ingress layer scalability fault tolerance
DNS
Routes clients to entry points. GeoDNS for multi-region routing — clients resolve to the nearest region. TTL controls propagation lag vs. failover speed.
read-heavy globalmulti-region
Load balancer — L4 vs L7
L4 (TCP): raw throughput, minimal overhead. L7 (HTTP): content-aware routing, sticky sessions, SSL termination, path-based routing. Almost always L7 for web services. Health checks remove dead nodes automatically.
alwaysL4 for raw perfL7 for HTTP apps
API gateway
Single entry point for microservices. Handles rate limiting, auth, request routing, response caching, SSL termination, request/response transformation. Adds latency (~1–5ms), justified when you have 3+ downstream services.
microservicessingle monolith
CDN
Edge cache for static assets and cacheable API responses. Reduces origin load by 80–95% for media-heavy apps. Cache-Control headers drive TTL. Invalidation is the hard part — use versioned URLs where possible.
static / mediaglobal users
Compute layer scalability fault tolerance
Stateless services
Any replica handles any request — session state lives in external cache (Redis). Enables horizontal scaling and easy failover. Default design target. If you find yourself needing sticky sessions, reconsider.
default choice
Worker pool / async processing
Consumes queue tasks independently from the web tier. Scale workers by queue depth. Good for CPU-heavy, slow, or unreliable work — video encoding, email sending, ML inference, report generation.
async workloadsslow ops
Serverless / FaaS
Scales to zero, billed per execution. Ideal for spiky or infrequent workloads. Cold start latency (50ms–2s) is the main tradeoff. Not suitable for latency-sensitive synchronous APIs.
event-drivenspikylow-latency APIs
Caching scalability durability
Cache-aside (lazy loading)
App checks cache → on miss, loads from DB and populates cache. Most common pattern. Cache only holds what's been requested — no wasted memory. Cache stampede risk on cold start or TTL expiry; use probabilistic early expiration or mutex locks.
read-heavyRedis, Memcached
Write-through
Writes go to cache AND DB atomically. Cache is always fresh — no stale reads. Higher write latency. Good when reads immediately follow writes (user profile updates, settings).
read + write balance
Write-behind (write-back)
Write to cache, flush to DB asynchronously. Lowest write latency. Risk: data loss if cache node dies before flush. Only use if you can tolerate that window.
very write-heavyfinancial / critical data
Cache invalidation strategies
TTL-based: simple, eventual consistency. Event-driven: invalidate on write events via pub/sub. Versioned keys: never invalidate — new key = new version. Versioned keys are the safest at scale.
hardest problem in CS
Message queues & streaming scalability fault tolerance durability
Message queue
Decouples producer from consumer. At-least-once delivery — consumer must be idempotent. Message deleted after acknowledgment. Good for task distribution: one task, one worker. SQS, RabbitMQ.
async jobsdecoupling
Event stream (Kafka)
Ordered, replayable, retained log. Multiple consumer groups read independently at their own offset. Retained for days/weeks. Good for audit log, fan-out, real-time analytics, CDC (change data capture).
fan-outaudit loganalytics
Pub/sub
Publisher emits to a topic; each subscriber gets a copy. No retention — subscribers must be online. Good for notifications, live updates, cache invalidation broadcasts. SNS, Google Pub/Sub.
notificationsfan-out
Dead letter queue (DLQ)
Receives messages that failed processing after N retries. Essential for debugging and preventing poison-pill messages from blocking a queue. Always set one up alongside any queue.
always with queues
Storage scalability durability fault tolerance
Relational DB (SQL)
ACID transactions, enforced schema, complex joins. Read replicas for scaling reads. Vertical scaling first, then read replicas, then sharding as a last resort. Use when correctness and consistency are critical — financial records, user accounts, relational data.
financialrelational datamassive write throughput
NoSQL — which type?
Key-value (Redis, DynamoDB): simple lookups, O(1). Document (MongoDB): flexible schema, nested objects. Wide-column (Cassandra): write-optimized, time-series, huge scale. Graph (Neo4j): relationship traversal. Choose by access pattern, not familiarity.
high write throughputflexible schema
Object storage (S3-style)
Infinitely scalable, cheap, 11-nines durability. Blob storage for media, backups, data lake. Not suitable for transactional data or random small writes. Pair with a CDN for reads.
media / filesbackupstransactional data
Search index (Elasticsearch)
Inverted index for full-text search, faceting, relevance ranking. Synced from primary DB via Kafka CDC or change streams. Never use as primary store. Keep in sync; accept slight lag.
full-text searchautocomplete
Replication & partitioning scalability durability
Leader-follower replication
One write node (leader), N read replicas (followers). Replication lag introduces eventual consistency on reads. Scale reads horizontally without touching write path. Automatic failover promotes a follower.
read-heavy
Database sharding
Horizontal partitioning across nodes. Hash sharding for even distribution; range sharding for ordered scans. Beware hotspots (celebrity problem). Cross-shard queries are painful — design to avoid them. Last resort after read replicas and caching.
write throughputhuge datasetscross-shard joins
Consistent hashing
Ring-based key assignment. Adding/removing a node only moves ~1/N of keys, not a full reshuffle. Virtual nodes (vnodes) even out load distribution. Standard for distributed caches and DHTs.
distributed cacheDHT
Quorum reads/writes
W + R > N guarantees you read your own write. Typical: N=3, W=2, R=2. Trade latency for consistency. Cassandra, DynamoDB use this model. Strong consistency at the cost of write latency.
distributed storesW+R > N for consistency
Fault tolerance & reliability fault tolerance durability
Circuit breaker
Stops cascading failures. Three states: Closed (normal), Open (fail fast, don't call downstream), Half-open (probe if recovered). Opens when error rate exceeds threshold. Wrap every downstream call.
microservicesexternal APIs
Retry + exponential backoff + jitter
Retry transient failures with exponential backoff. Jitter (randomized delay) prevents thundering herd — all clients retrying simultaneously. Cap max retries (3–5). Idempotency key required for safe mutation retries.
transient failures
Idempotency keys
Client-generated UUID per operation. Server deduplicates by key — safe to retry without double-execution. Store results keyed by idempotency key for 24h. Critical for payments, order creation, any state-changing op.
paymentsmutations
Health checks (liveness + readiness)
Liveness: is the process alive? Readiness: is it ready to serve traffic? Load balancer routes away from unready nodes. Kubernetes uses both. Readiness prevents traffic to nodes still warming up their cache.
always
Multi-region deployment
Active-active: traffic in all regions, harder consistency (conflict resolution). Active-passive: one region serves, other on standby (simpler, RTO ~minutes). Choose based on availability SLA and consistency requirements.
high availability SLAactive-passive simpler
Write-ahead log (WAL)
Every write appended to sequential log before applied to data structure. Enables crash recovery — replay log on restart. Foundation of all durable storage (Postgres, Kafka, RocksDB). Also enables replication.
any persistent store
Bulkhead pattern
Isolate thread pools or connection pools per downstream service. One slow dependency can't exhaust shared resources. Named after ship compartments — one flood doesn't sink the ship.
multi-tenantshared infra
Saga pattern
Distributed transactions without 2PC. Chain of local transactions; each publishes an event. Compensating transactions roll back on failure. Two styles: choreography (events) vs orchestration (central coordinator).
distributed transactions2PC
Rate limiting & back-pressure scalability
Token bucket
Tokens refill at steady rate up to capacity. Request costs 1 token. Allows bursts up to bucket size. State: {tokens, last_refill_time} — O(1) per user. Most common for API rate limits. Used by AWS, Stripe.
API rate limitsbursty traffic
Leaky bucket
Requests queue; output drains at fixed rate regardless of input. Excess dropped. Guarantees smooth downstream throughput — can't burst. Adds latency. Use to protect slow downstream services.
traffic shapingprotect slow downstream
Sliding window log
Track timestamps of requests; count those within window. No boundary spike problem (vs fixed window). More accurate but O(n) memory per user. Redis sorted set: score = timestamp, range query for window.
strict fairnessRedis sorted set
Back-pressure
Producer slows when consumer can't keep up. Prevents memory blowup and cascading overload. Signal via: HTTP 429, queue depth metric triggering auto-scaling, TCP flow control. Design consumers to emit back-pressure explicitly.
streamingasync pipelines
Requirements to clarify in interviews requirements
Always clarify requirements before designing. The first 5 minutes of a system design interview should surface the constraints that determine every component choice. These are the axes that matter most.
Scalability
How many users / requests per second at peak? What's the growth rate? Defines whether a single DB suffices or you need sharding, whether one region is enough, what cache hit rate you need.
Ask: DAU, peak RPS, data volume
Availability (nines)
99.9% = 8.7 hr/yr downtime. 99.99% = 52 min/yr. Drives multi-AZ vs multi-region, active-active vs active-passive, RTO/RPO requirements. Higher nines = exponentially more complexity and cost.
Ask: availability SLA, acceptable downtime
Consistency
Strong: every read sees the latest write (expensive). Eventual: reads may be stale (cheap). Causal: reads see writes in causal order. Most social features accept eventual; financial ops require strong. State this explicitly — don't let the interviewer infer it.
Ask: can users see stale data? For how long?
Durability
Can data ever be lost? What's the acceptable RPO (recovery point objective)? Drives replication factor, sync vs async writes, backup frequency. Financial data: zero loss. Social likes: losing a few is acceptable.
Ask: what's the cost of data loss?
Latency & performance
p50 vs p99 vs p999. The difference between "fast on average" and "never slow." Identify the latency-sensitive path. Async offload anything that doesn't need to be synchronous. SLAs are usually expressed as p99 < N ms.
Ask: p99 latency target, sync vs async
Throughput
Reads/writes per second at scale. Read-heavy vs write-heavy shapes the entire architecture — read-heavy → caches + replicas; write-heavy → sharding + async queues. 80/20 read-write split is common but verify.
Ask: read vs write ratio
Security & auth
AuthN: who are you? (JWT, session token, OAuth). AuthZ: what can you do? (RBAC, ABAC). API key management for service-to-service. Rate limit per identity, not just per IP. Especially important for payment and user-data APIs.
Ask: auth requirements, public vs private API
Observability
Logging (structured, searchable), metrics (latency, error rate, saturation — RED method), tracing (distributed trace per request). How do you know the system is healthy? What pagers you at 3am? Proactively mentioning this impresses interviewers.
Logs + metrics + traces = pillars
Data model & access patterns
What queries must be fast? What's the read/write access pattern? This determines DB choice more than anything else. "Point lookup by ID" → key-value. "Fan-out with joins" → SQL. "Time-ordered by user" → wide-column. Model queries first, schema second.
Ask: primary query patterns
Cost & efficiency
Storage cost, compute cost, egress cost. At FAANG scale, architecture IS cost. Choosing async over sync, batch over streaming, S3 over EBS, spot instances for workers — all cost-motivated. Mention cost when justifying component choices.
Egress is often the surprise
Compliance & data residency
GDPR: right to erasure, data stays in EU. HIPAA: PHI encryption, audit logs. PCI-DSS: card data isolation. Data residency can force multi-region to be mandatory, not optional. Always ask in any design involving user PII.
Ask: PII, geographic requirements
Fault tolerance requirements
What failure modes must the system survive? Single node, AZ, region? What's the RTO (time to recover)? What degraded mode is acceptable — read-only? Cached responses? This drives whether you need active-active or active-passive.
Ask: RTO, RPO, acceptable degraded modes
CAP theorem requirements
CP — Consistent + Partition-tolerant
Use when correctness > availability. System may be unavailable during network partition but will never return stale data. Banking, inventory, leader election, distributed locks.
Examples: HBase, Zookeeper, etcd, MongoDB (w: majority)
AP — Available + Partition-tolerant
Use when uptime > strict consistency. Returns possibly stale data during partition. Social feeds, shopping carts, DNS caches, product catalogs.
Examples: Cassandra, DynamoDB (default), CouchDB, Riak
Interview nuance: Most real systems accept eventual consistency for reads with strong consistency on writes to a primary. Say this explicitly — "I'll use leader-follower replication, accepting a short replication lag on reads from replicas, which is acceptable for this use case because…" That sentence alone signals mature thinking.

Interview signal cues signals
"Design a system for millions of users"
Stateless services + horizontal scaling + load balancer + read replicas. Add caching layer. Mention read replicas before sharding — sharding is a last resort. Estimate: at 1K RPS/server, 100K RPS = 100 servers.
"How do you handle a viral post / traffic spike?"
CDN for static, cache-aside for hot data, queue to absorb write bursts, auto-scaling compute, back-pressure on queue consumers. Pre-warm caches for anticipated events.
"What happens if a service goes down?"
Circuit breaker + retry with jitter, health checks in LB route away instantly, queue buffers in-flight work during outage, DB replicas cover read traffic. Critical path → multi-region active-passive.
"How do you avoid data loss?"
WAL, replication factor ≥ 3, only ack after quorum write, S3 versioning for blobs, idempotency keys for retries, scheduled backups with tested restore.
"How do you keep the feed / timeline fast?"
Pre-compute + fan-out-on-write for small follow counts. Fan-out-on-read for celebrities (pull hybrid). Cache in Redis sorted set keyed by user ID, score = timestamp.
"How do you implement search efficiently?"
Elasticsearch / Lucene inverted index. Sync from DB via Kafka CDC. Separate read path from write path. Mention relevance ranking, autocomplete via prefix tree if asked.
"How do you count unique views at scale?"
HyperLogLog for approximate unique counts (~2% error, O(1) memory). Kafka for event ingestion. Batch aggregation (Spark/Flink) for exact counts with higher latency. Pre-aggregate into time-bucketed counters in Redis.
"How would you design the payment flow?"
Idempotency key on every request. Saga pattern for distributed transaction (not 2PC). At-least-once queue with dedup. Audit log via WAL. Reconciliation job catches gaps. PCI-DSS data isolation.
"How do you know if your system is healthy?"
RED method: Rate (req/s), Errors (error%), Duration (latency). Structured logs → searchable (Datadog, ELK). Distributed tracing for slow request root cause. Alerts on p99 latency + error rate, not just avg.
"How do you handle multi-region consistency?"
State the tradeoff: active-active needs conflict resolution (last-write-wins, CRDTs, or application-level merge). Active-passive is simpler, RTO ~minutes. Most companies start active-passive, promote to active-active when they need it.
