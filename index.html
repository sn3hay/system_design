Cachingscalabilitydurability
Cache-aside (lazy loading)
App checks cache → on miss, loads from DB and populates. Most common pattern. Cache stampede risk on cold start — use mutex locks or probabilistic early expiration.
read-heavyRedis, Memcached
Write-through
Writes go to cache AND DB atomically. Cache always fresh — no stale reads. Higher write latency. Good when reads immediately follow writes.
read + write balance
Write-behind (write-back)
Write to cache, flush to DB async. Lowest write latency. Risk: data loss if cache node dies before flush. Only use if you can tolerate that window.
very write-heavyfinancial / critical data
Cache invalidation strategies
TTL-based: simple, eventual consistency. Event-driven: invalidate on write events. Versioned keys: never invalidate — new key = new version. Versioned keys are safest at scale.
hardest problem in CS

Ordered, replayable, retained log. Multiple consumer groups read independently. Good for audit log, fan-out, real-time analytics, CDC (change data capture).

Receives messages that failed processing after N retries. Essential for debugging and preventing poison-pill messages from blocking a queue. Always pair with any queue.

Relational DB (SQL)
ACID transactions, enforced schema, complex joins. Read replicas for scaling reads. Vertical scaling first, then read replicas, then sharding as last resort. Use when correctness and consistency are critical.
financialrelational datamassive write throughput
NoSQL — which type?
Key-value (Redis, DynamoDB): O(1) lookups. Document (MongoDB): flexible schema. Wide-column (Cassandra): write-optimized, time-series. Graph (Neo4j): relationships. Choose by access pattern.
high write throughputflexible schema
Object storage (S3-style)
Infinitely scalable, cheap, 11-nines durability. Blob storage for media, backups, data lake. Not suitable for transactional data or random small writes. Pair with a CDN for reads.

Replication & partitioningscalabilitydurability
Leader-follower replication
One write node (leader), N read replicas (followers). Replication lag introduces eventual consistency on reads. Scale reads horizontally. Automatic failover promotes a follower.
read-heavy
Database sharding
Horizontal partitioning across nodes. Hash sharding for even distribution; range sharding for ordered scans. Beware hotspots. Cross-shard queries are painful. Last resort after read replicas and caching.
write throughputhuge datasetscross-shard joins
Consistent hashing
Ring-based key assignment. Adding/removing a node only moves ~1/N of keys. Virtual nodes (vnodes) even out load distribution. Standard for distributed caches and DHTs.
distributed cacheDHT
Quorum reads/writes
W + R > N guarantees you read your own write. Typical: N=3, W=2, R=2. Trade latency for consistency. Cassandra, DynamoDB use this model.
distributed storesW+R > N for consistency
Fault tolerance & reliabilityfault tolerancedurability
Circuit breaker
Stops cascading failures. States: Closed (normal), Open (fail fast), Half-open (probe recovery). Opens when error rate exceeds threshold. Wrap every downstream call.
microservicesexternal APIs
Retry + exponential backoff + jitter
Retry transient failures with exponential backoff. Jitter prevents thundering herd. Cap max retries (3–5). Idempotency key required for safe mutation retries.
transient failures
Idempotency keys
Client-generated UUID per operation. Server deduplicates by key — safe to retry without double-execution. Store results for 24h. Critical for payments, order creation, any state-changing op.
paymentsmutations
Health checks (liveness + readiness)
Liveness: is the process alive? Readiness: is it ready to serve traffic? LB routes away from unready nodes. Readiness prevents traffic to nodes still warming up.
always
Multi-region deployment
Active-active: traffic in all regions, harder consistency. Active-passive: one region serves, other on standby, RTO ~minutes. Choose based on availability SLA.
high availability SLAactive-passive simpler
Write-ahead log (WAL)
Every write appended to sequential log before applied. Enables crash recovery — replay log on restart. Foundation of all durable storage (Postgres, Kafka, RocksDB). Also enables replication.
any persistent store
Bulkhead pattern
Isolate thread/connection pools per downstream service. One slow dependency can't exhaust shared resources. Named after ship compartments — one flood doesn't sink the ship.
multi-tenantshared infra
Saga pattern
Distributed transactions without 2PC. Chain of local transactions; each publishes an event. Compensating transactions roll back on failure. Choreography vs orchestration styles.
distributed transactions2PC
Rate limiting & back-pressurescalability
Token bucket
Tokens refill at steady rate up to capacity. Request costs 1 token. Allows bursts up to bucket size. State: {tokens, last_refill_time} — O(1) per user. Most common for API rate limits.
API rate limitsbursty traffic
Leaky bucket
Requests queue; output drains at fixed rate regardless of input. Excess dropped. Guarantees smooth downstream throughput — can't burst. Adds latency.
traffic shapingprotect slow downstream
Sliding window log
Track timestamps of requests; count those within window. No boundary spike problem vs fixed window. O(n) memory per user. Redis sorted set: score = timestamp.
strict fairnessRedis sorted set
Back-pressure
Producer slows when consumer can't keep up. Prevents memory blowup and cascading overload. Signal via: HTTP 429, queue depth metric, TCP flow control.
streamingasync pipelines

CAP theoremrequirements
CP — Consistent + Partition-tolerant
Use when correctness > availability. System may be unavailable during network partition but will never return stale data. Banking, inventory, leader election, distributed locks.
Examples: HBase, Zookeeper, etcd, MongoDB (w: majority)
AP — Available + Partition-tolerant
Use when uptime > strict consistency. Returns possibly stale data during partition. Social feeds, shopping carts, DNS caches, product catalogs.
Examples: Cassandra, DynamoDB (default), CouchDB, Riak
Interview nuance: Most real systems accept eventual consistency for reads with strong consistency on writes to a primary. Say this explicitly — "I'll use leader-follower replication, accepting a short replication lag on reads from replicas, which is acceptable for this use case because…" That sentence alone signals mature thinking.

Interview signal cuessignals
"Design a system for millions of users"
Stateless services + horizontal scaling + load balancer + read replicas. Add caching layer. Mention read replicas before sharding — sharding is a last resort. Estimate: at 1K RPS/server, 100K RPS = 100 servers.
"How do you handle a viral post / traffic spike?"
CDN for static, cache-aside for hot data, queue to absorb write bursts, auto-scaling compute, back-pressure on queue consumers. Pre-warm caches for anticipated events.
"What happens if a service goes down?"
Circuit breaker + retry with jitter, health checks in LB route away instantly, queue buffers in-flight work during outage, DB replicas cover read traffic. Critical path → multi-region active-passive.
"How do you avoid data loss?"
WAL, replication factor ≥ 3, only ack after quorum write, S3 versioning for blobs, idempotency keys for retries, scheduled backups with tested restore.
"How do you keep the feed / timeline fast?"
Pre-compute + fan-out-on-write for small follow counts. Fan-out-on-read for celebrities (pull hybrid). Cache in Redis sorted set keyed by user ID, score = timestamp.
"How do you implement search efficiently?"
Elasticsearch / Lucene inverted index. Sync from DB via Kafka CDC. Separate read path from write path. Mention relevance ranking, autocomplete via prefix tree if asked.
"How do you count unique views at scale?"
HyperLogLog for approximate unique counts (~2% error, O(1) memory). Kafka for event ingestion. Batch aggregation (Spark/Flink) for exact counts with higher latency.
"How would you design the payment flow?"
Idempotency key on every request. Saga pattern for distributed transaction (not 2PC). At-least-once queue with dedup. Audit log via WAL. Reconciliation job catches gaps. PCI-DSS data isolation.
"How do you know if your system is healthy?"
RED method: Rate, Errors, Duration. Structured logs → searchable. Distributed tracing for slow request root cause. Alerts on p99 latency + error rate, not just avg.
"How do you handle multi-region consistency?"
State the tradeoff: active-active needs conflict resolution (last-write-wins, CRDTs). Active-passive is simpler, RTO ~minutes. Most companies start active-passive, promote when needed.

Consistencycorrectness axis

Consistency answers: "if I write data right now, when does a subsequent read see it?" There are three models you must be able to name and reason about.

Strong consistency
Every read sees the most recent write. Requires coordination — every write confirmed by quorum before returning. High latency, lower availability.
Never staleSlower writes, lower availability
Use: bank balances, inventory, locks
Eventual consistency
Writes propagate async — replicas converge given no new writes. Reads may be stale for ms to seconds. Cheap, fast, highly available.
Fast, highly availableReads may be stale
Use: social feeds, likes, DNS, caches
Causal consistency
Reads respect causal order — if you wrote A then B, any reader who sees B also sees A. Middle ground for collaborative apps.
Logical order preservedComplex to implement
Use: messaging, collaborative docs
When does stale matter? Social feed 2s stale = invisible. Shopping cart stale = oversell. Bank balance stale = fraud / regulatory violation. The answer drives your model.
Interview line: "I'll use eventual consistency for the feed — a like count 2 seconds late has no business impact. But for the payment ledger, I need strong consistency with synchronous quorum writes. I'm explicitly choosing different models for different parts of the system."
Availabilityuptime axis

Availability = the fraction of time the system correctly responds to requests. Each nine costs exponentially more in complexity and money.

Planned vs unplanned downtime
Planned: deployments, maintenance, schema migrations. Mitigate with rolling deploys, blue-green, canary releases. Unplanned: crashes, partitions, cascading failures. Mitigate with circuit breakers, retries, redundancy.
What does "available" mean for your system?
A search engine returning cached results during a DB outage = available. A payment processor returning errors = unavailable. Define in terms of user-visible impact — degraded mode may count as available.
Interview move: "Clarify the SLA first. Then trace the single points of failure. Every component has at least one backup. LB health checks remove failed nodes in <10s. DB has a hot standby that auto-promotes. That gets me to 99.99%."
Latency & performancespeed axis

p50 is "fast on average." p99 is "almost never slow." p999 is "the worst 1-in-1000 request." SLAs are almost always p99 — average hides the long tail that destroys user experience.

p50 (median)
Half of requests are faster, half slower. Good for understanding typical experience. Useless for SLA commitments — a p50 of 20ms can coexist with a p99 of 5s.
p99
99% of requests complete within this time. The standard SLA target. 100ms p99 means 1 in 100 users waits >100ms. This is what you optimize for — forces you to eliminate outliers, not just averages.
p999
1-in-1000 worst case. At 1M RPS, 1000 requests/second hit p999. Often caused by GC pauses, cold cache, lock contention. Hard to eliminate, important to cap.
Finding the critical path
Draw the request flow. Identify every synchronous hop — each stacks latency. Async offload anything non-critical: sending email, analytics fan-out. User waits only for: auth check + DB read + response.
Optimization toolkit
Cache in front of DB (100µs vs 10ms)Read replica for read pathAsync queue for write fan-outCDN for static / predictable contentConnection pooling (avoid handshake cost)Premature optimization hides bugs
Interview move: "The latency-sensitive path here is auth → cache lookup → response. Targeting p99 < 50ms. Auth token validation in-process (no network hop). Cache hit eliminates the DB call. On cache miss, I'll use a read replica co-located with the app server to cut RTT."
Throughputvolume axis

Throughput = requests or bytes per second your system must handle. It's the number that forces architectural decisions: when does a single DB stop being enough?

Back-of-envelope math
Always estimate first. 10M DAU × 10 reads/day = 100M reads/day = 1,160 reads/sec. A single Postgres handles ~10K simple reads/sec. So one DB is fine. 1B DAU changes that math entirely.
Read vs write ratio
Most systems are read-heavy (10:1 to 100:1). Read-heavy → caches + read replicas. Write-heavy → distribute the write path (sharding, partitioned Kafka, async buffering). Always confirm the ratio.
Throughput levelWhat breaks firstSolution

1K–10K RPSDB read bottleneckRead replicas, cache layer
10K–100K RPSSingle-node computeHorizontal app scaling, LB, partitioned cache
100K+ RPSDB write bottleneckSharding, CQRS, async write queue
Interview move: "At 500K DAU with a 20:1 read-write ratio, I'm at ~1200 writes/sec and 24K reads/sec. A single DB handles the writes. Read replicas handle the reads. I don't need sharding at this scale — it would add complexity without benefit."
CAP theorem & PACELCtradeoff framework

CAP says: during a network Partition, choose between Consistency and Availability. PACELC extends it: even without a partition, there's a tradeoff between Latency and Consistency.

CAP — during a partition
CP systems: return error during partition — never stale data. Correct but unavailable.

AP systems: return possibly stale data during partition — always respond. Available but potentially wrong.

CA doesn't exist in distributed systems — partitions always happen.
PACELC — the fuller picture
During normal operation, every distributed system trades latency for consistency. Synchronous replication = higher write latency. Async replication = lower write latency, possible stale reads.

Most real decisions are PACELC not CAP — partitions are rare, but the latency/consistency tradeoff is constant.

Security & authtrust axis

AuthN (authentication) = who are you? AuthZ (authorization) = what can you do? They're separate concerns and should be separate systems.

JWT vs opaque tokens
JWT (stateless): self-contained, no DB lookup. Fast. Can't be invalidated before expiry — use short TTL (15min) + refresh token. Good for stateless microservices.

Opaque token (stateful): validated by DB/cache lookup. Can be revoked instantly. Slightly slower. Good when immediate revocation matters.
AuthZ models
RBAC (role-based): user has a role. Simple, coarse-grained. Good default.

ABAC (attribute-based): access based on attributes. Flexible, complex. Fine-grained enterprise access.

ReBAC (relationship-based): Google Zanzibar model. Good for social/collaborative systems.
API keys (service-to-service)
Long-lived secrets for programmatic access. Store hashed (never plaintext). Scope to minimum permissions. Rotate regularly. Rate limit per key, not per IP.
Rate limiting per identity
Per-user or per-API-key limits prevent abuse without blocking all users. Requires auth before rate limit check. Store counters in Redis keyed by user_id + window.
Transport & at-rest
TLS everywhere (including internal calls). Encrypt sensitive fields at rest. Key management via KMS. mTLS for service-to-service in zero-trust networks.
Interview move for payment APIs: "Auth is two layers: JWT for the user session (15-min expiry, refresh token in httpOnly cookie), and API key with HMAC signature for merchant integrations. Rate limited per merchant key at the API gateway. Card data never touches our servers — tokenized at the edge."
Observabilityvisibility axis

Three pillars: logs, metrics, traces. Mentioning this proactively before you're asked is a strong interview signal — it shows you've run systems in production.

Logs
Structured (JSON), not plaintext — searchable by field. Include: request ID, user ID, latency, status code, service name. Centralized in ELK / Datadog. Retention: hot 7 days, cold 90 days. Never log PII or secrets.
Metrics (RED method)
Rate — requests/sec per service
Errors — error % per endpoint
Duration — p50/p99 latency

Time-series in Prometheus/Datadog. Alert on error rate spike and p99 degradation — not CPU/memory alone.
Distributed tracing
Propagate a trace ID through every service hop. Reconstruct the full call tree for a slow request. Critical for microservices — you can't root-cause a 2s request without it. Jaeger, Zipkin, Datadog APM.
What to alert on
Error rate > 1% for 2 min — pagep99 latency > SLA threshold — pageQueue depth growing — warnDB replication lag > 5s — warnAlert on averages — hides tail latencyAlert on CPU/memory alone — noisy
Correlation is the goal
A spike in error rate should link to: the specific trace showing what failed, the log lines with the error, the deploy that happened 5 min ago. Without all three pillars correlated by request ID and timestamp, you're flying blind.
Interview move: "I'd add structured logging with a correlation ID propagated across services, RED metrics on every endpoint in Prometheus, and distributed tracing sampling 1% of requests (100% on errors). Alert on: error rate >0.5% sustained 2 min, p99 >200ms, queue depth >10K messages."

Denormalization vs normalization
SQL normalizes — good for writes, bad for reads (joins). NoSQL denormalizes — fast point reads, expensive updates. Choose based on read/write ratio and query shape. Reads dominating? Denormalize.
The hotspot problem
Partitioning by a monotonic key (timestamp, auto-increment ID) sends all new writes to one partition — a hotspot. Solution: add a random prefix (salting), or use hash-based partitioning.
Interview move: "Before picking a DB, I want to confirm the access patterns. 'Get user profile by ID' + 'get posts by user in time order' → DynamoDB with composite key (user_id, timestamp). O(1) user lookup and efficient time-range scans within a partition, no joins needed."
Cost & efficiencyeconomics axis

At FAANG scale, architecture is cost. Every component choice has a dollar sign. Mentioning cost tradeoffs signals that you think like an engineer at scale, not just a student.

Storage cost hierarchy

EBS ~$0.08–0.10/GB/mo — block, low latency
EFS ~$0.30/GB/mo — shared file
RDS ~$0.115/GB/mo + compute

Compute cost patterns
Spot/preemptible — 70% cheaper for batchServerless scales to zero — no idle costReserved instances — 40% off for predictable loadOver-provisioned persistent servers — common wasteSync calls blocking a thread — CPU waste
Egress is the surprise
AWS charges ~$0.09/GB leaving the region. At 1 PB/month = $90K/month in egress alone. CDN reduces origin egress. Keep analytics processing in the same region as data.
Async vs sync cost
Synchronous: thread held for entire duration. 100ms p99 × 1K RPS = 100 threads busy at all times. Async + queue: thread released immediately. 10× fewer threads for same throughput. Async is cheaper and more scalable.
Batch vs streaming cost
Streaming (Kafka, Kinesis): continuous, higher infra cost, real-time results. Batch (Spark, EMR): scheduled, cheap spot instances, minutes/hours delay. If hourly is fine, batch it — often 10× cheaper.
Interview move: "For the analytics pipeline, I'd separate hot and cold paths. Real-time metrics (p99, error rate) stream through Kafka — we need those instantly. Historical aggregations (DAU, retention cohorts) run as nightly Spark jobs on spot instances — no need to pay for streaming infra for something that runs once a day."
Compliance & data residencylegal axis

Compliance requirements are hard constraints that override architectural preferences. Know the three major frameworks and what they require technically.

GDPR (EU)
Data of EU residents must stay in EU. Right to erasure: delete all user data within 30 days — hard if you denormalized everywhere. Data minimization: only collect what you need.
Technical: EU-only region, soft-delete + periodic hard-delete job, PII catalog
HIPAA (US healthcare)
PHI encrypted at rest and in transit. Access logs mandatory. BAAs required with all vendors. No PHI in logs. Minimum necessary access principle.
Technical: field-level encryption, audit log table, KMS for keys
PCI-DSS (payments)
Cardholder data isolated in separate network segment. No raw card numbers stored — tokenize immediately. Most teams avoid this by using Stripe/Braintree.
Technical: tokenization at edge, network segmentation, no card data in your DB
Data residency in architecture
Forces multi-region to be mandatory, not a scaling choice. EU traffic must be handled by EU servers and stored in EU databases — cannot route to US for processing. Design region-isolated data planes from the start — retrofitting is extremely expensive.
Right to erasure challenge
Easy in normalized SQL: DELETE cascades. Hard in: denormalized NoSQL (user ID embedded everywhere), Kafka (immutable log), data warehouses, backups. Solution: pseudonymization — replace PII with a token; deletion just removes the token mapping.
Interview move: "I'd ask up front: do we have EU users? If yes, I need EU-region data isolation from day one — retrofitting this is extremely expensive. I'd design for right-to-erasure using pseudonymization: store a mapping of user_id → random token, embed the token everywhere else. Erasure just deletes the mapping row."
